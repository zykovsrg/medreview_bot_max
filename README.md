# Medreview Bot For MAX

Отдельная версия бота для мессенджера MAX, собранная на базе уже существующего Telegram-бота, но без изменения Telegram-проекта.

## Что переиспользовано

- `app/google_clients.py` — интеграция с Google Sheets / Google Docs через Apps Script.
- `app/doc_parser.py` — разбор структуры документа.
- `apps_script/Code.gs` — текущий Apps Script-контракт.
- SQLite-хранилище и логика напоминаний.
- Общая бизнес-логика сценария проверки статьи.

## Что заменено под MAX

- транспорт и обработка событий;
- кнопки и навигация;
- отправка медиа и иллюстраций;
- пересылка файлов и голосовых сообщений редактору;
- режим запуска `polling/webhook`.

## Важные отличия от Telegram

- в MAX используются inline-кнопки внутри сообщений, а не Telegram reply keyboard;
- reply в MAX есть, но официальные материалы подтверждают именно ответ на сообщение целиком;
- поэтому точную цитату выделенного фрагмента, как в Telegram, нельзя гарантировать один в один без живой проверки на реальном клиенте MAX;
- комментарий-ответ на конкретное сообщение бот сохранит.

## Рекомендуемый режим запуска

- для локальной разработки: `polling`;
- для продакшена: `webhook`, потому что официальная документация MAX рекомендует именно его для постоянной работы.

## Установка

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
python -m app.main
```

## Docker-запуск

Если удобнее запускать на сервере в контейнере:

```bash
cp .env.example .env
docker compose up -d --build
```

Это готовый базовый запуск контейнера. Для режима `polling` этого достаточно. Для режима `webhook` сверху ещё нужен публичный `HTTPS`-адрес, потому что MAX отправляет webhook только по `HTTPS`.

## Быстрая проверка перед запуском

После заполнения `.env` можно сделать безопасную предварительную проверку:

```bash
python -m app.preflight
```

Эта команда:

- проверит, что обязательные переменные заполнены;
- покажет текущую конфигурацию без вывода секретов;
- попробует обратиться к вашему Google Apps Script и убедиться, что он отвечает.

## Если MAX ругается на SSL

Если при запуске появляется ошибка вроде `CERTIFICATE_VERIFY_FAILED` или `self-signed certificate in certificate chain`,
это обычно значит, что сеть, прокси, антивирус или корпоративный фильтр подменяют сертификат.

Сначала лучше попробовать безопасный вариант:

- запустить с другой сети, например с мобильной точки доступа;
- или добавить доверенный сертификат вашей сети в систему/Python.

Если нужно быстро проверить бота именно в этой сети, можно временно отключить SSL-проверку только для запросов к MAX:

```env
MAX_SSL_VERIFY=false
```

Это временный компромисс. Для постоянной работы безопаснее вернуть:

```env
MAX_SSL_VERIFY=true
```

Если у вас есть путь к доверенному корпоративному сертификату, можно вместо отключения проверки указать его:

```env
MAX_CA_BUNDLE=/полный/путь/к/certificate.pem
```

## Подготовка к хостингу

Для простого первого размещения я рекомендую такой путь:

- локально тестировать в `polling`;
- первый серверный запуск тоже сделать в `polling`, если у вас ещё нет домена и HTTPS;
- после этого, когда будет домен и HTTPS, переключить сервер на `webhook`;
- держать проект в отдельной папке и отдельном контейнере, не смешивая его с Telegram-ботом.

Самый простой безопасный путь для первого деплоя:

```env
MAX_DELIVERY_MODE=polling
```

Такой режим не требует публичного адреса и обычно проще всего заводится на VPS.

Минимальные переменные для сервера:

```env
MAX_DELIVERY_MODE=webhook
MAX_WEBHOOK_HOST=0.0.0.0
MAX_WEBHOOK_PORT=8080
MAX_WEBHOOK_PUBLIC_URL=https://ваш-домен/webhook/
MAX_WEBHOOK_PATH=/webhook/
MAX_WEBHOOK_SECRET=придумайте-секрет-без-пробелов
```

`MAX_WEBHOOK_PATH` можно не заполнять вручную: если он пустой, бот сам возьмёт путь из `MAX_WEBHOOK_PUBLIC_URL`.

Что происходит при таком запуске:

- бот поднимает HTTP-сервер;
- reverse proxy или балансировщик принимает `HTTPS` снаружи и проксирует его на внутренний `http://бот:8080/`;
- при старте сам подписывает MAX на ваш `MAX_WEBHOOK_PUBLIC_URL`;
- входящие webhook-запросы дополнительно проверяются по заголовку `X-Max-Bot-Api-Secret`;
- для быстрой проверки доступен `GET /healthz`.

Если вы пока не хотите связываться с публичным HTTPS-адресом, можно запускать бот на VPS в `polling`. Это проще как первый деплой. Когда сервер стабильно заработает, можно спокойно перевести его на `webhook`.

## Источник Google-данных

Проект по умолчанию остаётся в той же модели:

- Python-бот не ходит в Google напрямую через OAuth;
- бот общается с опубликованным Google Apps Script Web App;
- Apps Script читает Google Sheets / Google Docs и возвращает JSON.

## Куда уходят отчёты редактору в MAX

В MAX лучше не привязываться к логину вида `@username`, потому что он может отсутствовать.
Поэтому в проекте сделана другая механика:

- откройте бота из того аккаунта MAX, куда должны приходить итоговые отчёты и комментарии врачей;
- отправьте команду `/register_report_chat`;
- после этого бот запомнит этот чат как получателя отчётов;
- проверить текущую привязку можно командой `/report_chat_status`;
- если нужно сменить получателя, просто откройте бота из другого аккаунта MAX и снова отправьте `/register_report_chat`.
